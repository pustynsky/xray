# Changelog


### Documentation (DOC-01: trigram rebuild locking discipline)

- **`docs/architecture.md` ŌĆö trigram lazy-rebuild section now matches the actual `ensure_trigram_index` implementation.** Pre-edit the doc claimed "rebuilt from scratch (~200ms)" with no mention of the locking strategy, which made it look like the rebuild blocks all readers under a write-lock for the duration. The current code (`src/mcp/handlers/grep.rs::ensure_trigram_index`, landed earlier this year) builds the new `TrigramIndex` under the **read** lock (concurrent `xray_grep` / `xray_callers` queries continue without contention), then takes the write-lock only to swap the pointer and flip the dirty flag (microseconds), with a double-checked `trigram_dirty` re-test inside the write critical section to drop duplicate work if two readers raced into rebuild. Updated the "Watcher Integration (Lazy Rebuild)" subsection to spell this out, added a small ASCII diagram of the read-lock / write-lock / read-lock sequence, and updated the "Lazy / On-Demand Rebuild" table row so it no longer carries the misleading "~200ms" absolute. The rebuild itself is still the dominant cost of the request that triggers it; what changed (and what the doc now reflects) is that other readers are no longer stalled by it. No code change.

- **`xray_callers` `interface_vias_caveat` ŌĆö no longer merges interface facts across unrelated same-short-name concrete types** (closes Problem 1 of `user-stories/xray-advisory-hints-namespace-collision-and-yml-gap.md`). Pre-fix, `interface_vias_caveat(class=Foo, ŌĆ”)` resolved `class` through `name_index.get("foo")` and aggregated `base_types` across **every** matching `Class | Struct | Record` entry ŌĆö so a codebase with `A.Foo : IFoo` and `B.Foo : IBar` got a hint claiming the targeted `Foo` "implements interface(s): `IFoo`, `IBar`" and recommending `class=IFoo` (which may belong to the OTHER `Foo`). That presented merged facts as if they apply to the selected class. **Fix**: collect the matching concrete defs first; the single-match path keeps the original wording verbatim; the N>1 path switches to ambiguity-aware wording that (a) explicitly announces "{N} types named `Foo` exist", (b) lists each candidate as `Parent.Name (file)` so the caller can disambiguate by inspection, (c) reports the *union* of interfaces with an explicit "some may not apply to the type you targeted" caveat, (d) drops the concrete `class={Iface}` recipe (the suggested interface might belong to a different `Foo` than the user targeted). N>1 with no interfaces returns `None` (nothing useful to suggest). **Tests** in `src/mcp/handlers/advisory_hints.rs`: `interface_vias_emits_ambiguity_notice_for_duplicate_short_name` (two `Foo` in different namespaces ŌåÆ hint contains "2 types named `Foo`", lists both `Ns.A.Foo (a/Foo.cs)` / `Ns.B.Foo (b/Foo.cs)`, surfaces `IFoo` + `IBar` as union, contains "some may not apply", does NOT contain `class=IFoo` / `class=IBar`), `interface_vias_returns_none_when_ambiguous_and_no_interfaces` (N>1 + zero interfaces ŌåÆ no hint). New test helper `build_index_with_parents` (with `DefSpec` type alias to satisfy `clippy::type_complexity`) populates `parent` and per-def `file_id` so the `Parent.Name (file)` candidate strings can be asserted.

- **`value_source_hint` ŌĆö suggested grep extension list now includes `yml`** (closes Problem 2 of the same story). The hint emits `xray_grep terms='ŌĆ”' ext='xml,json,config,yaml,manifestxml'` as a ready-to-paste command. Helm charts, GitLab CI, GitHub Actions workflows, k8s manifests, Azure Pipelines, and Ansible playbooks all use `.yml` (NOT `.yaml`) routinely ŌĆö so the omission caused silent false negatives in exactly the scenario the hint exists to help with. **Fix**: one-line change to `ext='xml,json,config,yaml,yml,manifestxml'`. The existing `value_source_hint_fires_for_property_with_attribute_literal` test was updated to pin the new ext list (`ext='xml,json,config,yaml,yml,manifestxml'`) with a comment noting why `yml` is mandatory.

- **`xray_edit` Mode A ŌĆö blank-line-only files no longer have a phantom addressable second line; line-count semantics now agree with `count_lines()` across all files** (closes `user-stories/xray-edit-blank-line-only-file-line-count-drift.md`). The 2026-04-23 hardening bundle added a carve-out for the empty-file case (`if lines == [""] { 0 } else { lines.len() }`) but left the adjacent blank-line-only case (`"\n"` ŌåÆ `["", ""]`, len 2) reporting `originalLineCount: 1` while Mode A range validation saw 2 addressable lines ŌĆö REPLACE 1..2 / DELETE 2 silently succeeded against a phantom slot. The same off-by-one applied to every file ending in `\n` (e.g. `"a\n"` ŌåÆ `["a", ""]`, len 2 ŌåÆ Mode A allowed REPLACE on phantom line 2, while `count_lines` and the response said "1 line"). The earlier carve-out closed only the `""` corner; this generalization closes the whole class. **Fix**: replace `let line_count = if lines == [""] { 0 } else { lines.len() }` with `let line_count = if lines.last() == Some(&"") { lines.len().saturating_sub(1) } else { lines.len() }` ŌĆö same rule as `count_lines()` (drop trailing-empty sentinel from `split('\n')`). INSERT mode (`startLine: 1, endLine: 0`) is unaffected and remains the canonical write-into-empty path. **Test update**: `test_mode_a_insert_at_end_of_file` previously documented the OLD off-by-one explicitly (passed `startLine: 4, endLine: 3` against `"line1\nline2\n"` because `split('\n')` had 3 elements); rewritten to use the human-aligned coordinates (`startLine: 3, endLine: 2`) with a comment explaining why the phantom slot is gone. **New tests** in `src/mcp/handlers/edit_tests.rs`: `test_blank_line_only_file_lf_treated_as_one_line` (`"\n"`: REPLACE 1..2 errors with "out of range"), `test_blank_line_only_file_crlf_treated_as_one_line` (`"\r\n"`: same), `test_blank_line_only_file_round_trip_line_counts_agree` (`expectedLineCount: 1` + REPLACE 1..1 succeeds ŌĆö addresses the only legitimate addressable line).

- **`xray_edit` ŌĆö out-of-range `occurrence` now reports the canonical "found only N time(s)" diagnostic BEFORE `expectedContext` validation runs** (closes `user-stories/xray-edit-expected-context-masks-occurrence-out-of-range.md`). Pre-fix `apply_literal_replace` clamped `target_idx` to 0 for out-of-range `occurrence` "so the context check doesn't panic before the canonical error fires below" (per the inline comment) ŌĆö but then immediately ran `check_expected_context(positions[0], ŌĆ”)`. If the supplied context disagreed with the *first* match's surroundings, the user got `Expected context ŌĆ”` for a match they never targeted, masking the more fundamental "you targeted occurrence 99 but only 2 matches exist" error. Code contradicted its own comment. **Fix**: hoist the `if edit.occurrence > count ŌåÆ Err("Occurrence N requested ŌĆ”")` check out of the per-mode (flex_re vs. literal) match arms and place it BEFORE the `expectedContext` block. Both the flex-regex and the literal branches now share the same single guard, and the inline `target_idx` calc inside the context block becomes infallible (`if occurrence == 0 { 0 } else { occurrence - 1 }` ŌĆö out-of-range cannot reach here). **Test** in `src/mcp/handlers/edit_tests.rs::test_occurrence_out_of_range_reported_before_expected_context_mismatch` (file with 2 `Foo` matches, `occurrence: 99 + expectedContext: "context that does not exist anywhere"` ŌåÆ asserts the response contains `Occurrence 99 requested` AND `found only 2 time(s)` AND does NOT contain `Expected context`, file unchanged on disk).

- **`xray_grep dir=<file>` auto-convert ŌĆö uses exact-basename match, not substring (no more sibling leak)** (closes follow-up `docs/user-stories/todo_approved_2026-04-24_grep-dir-file-autoconvert-substring-sibling-leak.md`). ### Bug Fixes (edit-correctness bundle 2026-04-24)

- **`xray_edit` ŌĆö `expectedContext` validates the *targeted* `occurrence`, not always the first match** (closes follow-up `docs/user-stories/todo_approved_2026-04-24_flex-expected-context-occurrence-safety-gap.md`). Pre-fix, `apply_literal_replace` ran `check_expected_context(result, search_result.positions[0], first_len, ctx_text)` unconditionally ŌĆö even when the user explicitly passed `occurrence: 2` to target the second match with disambiguating context. Two failure modes: (a) caller's intended replacement at occurrence N would be REJECTED by the gate failing on occurrence 1's surroundings (false-negative, "expected context not found" with the wrong line number); (b) symmetric false-positive ŌĆö caller targeting `occurrence: 2` with WRONG context (matching occurrence 1) would silently PASS the gate and replace the second match anyway, since the gate never looked at the targeted match. **Fix**: derive `target_idx` from `edit.occurrence` (`0 ŌåÆ 0` for replace-all preserves historical first-match contract; `n ŌåÆ n-1` for 1-indexed nth-match; out-of-range clamps to 0 so the canonical "found only N time(s)" error fires below). Use `flex_match_lens[target_idx]` when present, otherwise `match_len`. **Tests** in `src/mcp/handlers/edit_tests.rs`: `test_expected_context_validates_targeted_occurrence_not_first` (occurrence 2 + 2nd-match context succeeds, only 2nd `Foo` replaced), `test_expected_context_rejects_targeted_occurrence_with_wrong_context` (occurrence 2 + 1st-match context rejected, file unchanged), `test_expected_context_occurrence_zero_replace_all_uses_first_match_context` (back-compat guard for replace-all). Test bodies use >5 lines between the two matches because `check_expected_context` searches a ┬▒5-line window ŌĆö same-window matches can't be disambiguated and aren't the bug being fixed.

- **`xray_edit` multi-file ŌĆö per-file `contentIndexUpdated` mirrors actual batch outcome instead of "always true"** (closes follow-up `docs/user-stories/todo_approved_2026-04-24_sync-reindex-multifile-poisoned-lock-response-integrity.md`). ### Features

- **Advisory hints for `xray_callers` and `xray_definitions` ŌĆö surface known AST blind spots so the LLM agent cross-checks before drawing conclusions** (closes `user-stories/xray-response-hints-for-incomplete-results.md`; promoted from real session post-mortem 2026-04-23 where an agent silently built conclusions on an incomplete `xray_callers` result because `class=Foo` filter excluded DI-injected `IFoo` interface call sites). Three new shape-based hints, all in the new `src/mcp/handlers/advisory_hints.rs` module:
  1. **`xray_callers` interface-vias caveat** ŌĆö when `class=Foo` is passed and the indexed `Foo` declares one or more interface base types (looked up via `name_index` + `kind == Interface`), the response now includes an `advisories[]` entry naming every detected interface and suggesting `class=IFoo` (or no `class=`) to include interface-receiver call sites. Generic suffixes are stripped (`IRepository<T>` ŌåÆ `IRepository`) before lookup. Skips false positives where the base type is a class, not an interface.
  2. **`xray_callers` low-count cross-check hint** ŌĆö when `1..=3` callers are returned **AND the result was not truncated**, the response includes an advisory naming the asymmetry between AST-aware `xray_callers` (which honours receiver type and may miss DI-injected, dynamic-dispatch, or hard-to-infer-`var` call sites) and text-only `xray_grep`, with a copy-pasteable `xray_grep terms='MethodName' countOnly=true` command to validate completeness. Threshold deliberately excludes `0` (covered by the existing nearest-match `hint`) and `>3` (where the cost-benefit of cross-check flips). **Truncation guard**: the hint is suppressed when `summary.truncated` (hit `maxTotalNodes` cap) or `summary.perLevelTruncated` (hit `maxCallersPerLevel` cap) is set ŌĆö otherwise a method with 5 real callers capped at `maxCallersPerLevel=2` would falsely emit "Found 2 callersŌĆ”" and mis-attribute the cap to AST blind spots. Wired identically in both single-method and multi-method (batch comma-separated) paths; regression tests `test_advisory_low_count_suppressed_when_truncated` and `test_advisory_low_count_emitted_in_multi_method_batch` guard against single/multi drift.
  3. **`xray_definitions` value-source hint** ŌĆö Property/Field whose `attributes[]` contain at least one double-quoted string literal now emits a `valueSourceHint` field with the extracted literals as ready-to-grep keys and a `xray_grep terms='key1,key2' ext='xml,json,config,yaml,manifestxml'` command. **Shape-based heuristic, deliberately broad**: detection requires only (a) `kind == Property | Field` and (b) at least one extractable string literal across the attribute set ŌĆö ALL string literals from ALL attributes are surfaced as grep keys (not just the "first argument" of the first attribute), and there is no body/initializer check (the index does not store body for Property/Field). The framing `"If any attribute binds to external configuration..."` keeps the hint honest for non-binder attributes like `[Display("Name")]` ŌĆö it never claims the attribute IS a binder, only suggests where to look IF it is. **No hardcoded attribute name list** (deliberately rejected during design ŌĆö every team has custom binders like `[Knob]`, `[FromManifest]`, `[Tenanted]`, and any allow-list goes stale). Catches `[ConfigurationProperty(...)]`, `[FromKeyVault(...)]`, `[AppSetting(...)]`, `@Value("${...}")`, `#[serde(rename = "...")]` and equivalents in any framework / language with no per-framework code.

  All three are pure helpers (no I/O, no allocation outside the hint string) with 27 unit tests in `advisory_hints::tests` covering positive cases, negative cases, and edge conditions (empty literals, escaped quotes, unterminated literals, generic-suffix stripping, base-class-not-interface, threshold boundaries 0/1/3/4, plural/singular). 5 integration tests in `callers_tests.rs` exercise the wiring end-to-end through `handle_xray_callers` (single-method positive + interface-vias positive/negative + truncation suppression + multi-method batch). Tool descriptions for `xray_callers` and `xray_definitions` in `tool_definitions()` document the new `advisories[]` and `valueSourceHint` response fields so MCP clients see them at schema-discovery time, not just from runtime probing. The hints follow the principle "xray is an observation-divider, not a classifier" ŌĆö they always frame as "if X, then try Y", never claim "X is true".

- **`xray_grep` phrase search ŌĆö line-level posting intersection (Tier B) + deterministic tie-break** (follow-up to the Tier-A parallelisation entry below). Two related changes inside `src/mcp/handlers/grep.rs`. (1) `collect_phrase_matches` no longer intersects phrase tokens at the file level only and then re-scans every candidate file from disk.
